# mcx-poker 项目人话导览

这份文档不是设计文档，也不是 API 手册。它的目标是用人话解释：这个项目里每个主要目录是干什么的，你要改功能时大概应该去哪里看。

## 根目录先怎么看

项目根目录大概可以分成几类东西：

```text
mcx-poker/
├── README.md
├── pyproject.toml
├── src/
├── tests/
├── doc/
├── scripts/
├── .venv/
├── .git/
├── .mypy_cache/
├── .pytest_cache/
├── .ruff_cache/
├── .gitignore
└── .DS_Store
```

人话版解释：

- `README.md`：给人看的入口说明。怎么安装、怎么运行、怎么测试，先看这里。
- `pyproject.toml`：给 Python 工具看的配置。依赖、pytest、mypy、ruff、命令入口都在这里。
- `src/`：真正的业务代码都在这里。
- `tests/`：自动化测试都在这里。每个核心模块基本都有对应测试。
- `doc/`：需求、设计、任务拆分、实现决策和这份人话说明都在这里。
- `scripts/`：预留给脚本的地方。现在基本是空的，只有 `.gitkeep` 保住目录。
- `.venv/`：本地虚拟环境，装依赖用的。不是业务代码。
- `.git/`：Git 自己用的目录，平时不用管。
- `.mypy_cache/`、`.pytest_cache/`、`.ruff_cache/`：类型检查、测试、代码检查工具留下的缓存。可以帮助下次运行更快，不是业务代码。
- `.gitignore`：告诉 Git 哪些文件不要提交。
- `.DS_Store`：macOS 自动生成的文件，和项目逻辑没关系。

## src：真正写代码的地方

核心代码在：

```text
src/mcx_poker/
```

可以把它理解成“这个扑克平台的后端和前端实现”。

现在里面主要有这些模块：

```text
src/
├── mcx_poker/
│   ├── engine/
│   ├── table/
│   ├── players/
│   ├── observation/
│   ├── history/
│   ├── delivery/
│   └── extensions/
└── mcx_Poker.egg-info/
```

人话版：

- `mcx_poker/`：项目真正的 Python 包，主要代码都在这里。
- `mcx_Poker.egg-info/`：打包工具生成的元信息，比如包名、依赖、入口点。平时不用手写它，也不用从这里理解业务。

### engine：牌局推进和规则边界

```text
src/mcx_poker/engine/
├── actions.py
├── validator.py
├── pokerkit_adapter.py
└── game_loop.py
```

人话版：

- `actions.py`：定义玩家能做什么动作，比如弃牌、过牌、跟注、加注到多少、全下。
- `validator.py`：动作提交给规则引擎前，先在我们自己这里检查一遍。
- `pokerkit_adapter.py`：唯一直接接触 PokerKit 的地方。PokerKit 管真正的扑克规则、发牌、下注合法性和结算。
- `game_loop.py`：一手牌怎么往前走。轮到谁、问谁要动作、校验动作、提交动作、记录事件、结算，都在这里串起来。

最重要的边界是：除了 `pokerkit_adapter.py`，其他模块都不应该直接 import PokerKit。

### table：长期存在的牌桌

```text
src/mcx_poker/table/
└── manager.py
```

人话版：

- 这里管理“桌子”本身。
- 固定 6 个座位。
- 谁坐在哪个座位。
- 每个人还有多少筹码。
- 当前 button 在哪。
- 当前是不是有一手牌在打。
- 一手结束后怎么更新筹码、移动 button。

它不负责判断牌大牌小，也不负责某个动作是否合法。那些归 PokerKit 和 engine。

### players：玩家从哪里来

```text
src/mcx_poker/players/
└── controllers.py
```

人话版：

- 这里定义“玩家控制器”。
- 不管玩家是真人、bot、未来 LLM，Game Loop 只认一个接口：给你 Observation，你返回 Action。
- `HumanPlayerController` 会等前端通过 WebSocket 提交动作。
- `BotPlayerController` 会从合法动作里选一个动作，用来跑通本地闭环。

这里不应该偷看 PokerKit 内部状态，也不应该偷看对手底牌。

### observation：每个玩家能看到什么

```text
src/mcx_poker/observation/
└── builder.py
```

人话版：

- 这里负责生成“某个玩家视角下的信息”。
- 自己的底牌可以看到。
- 公共牌可以看到。
- 对手未公开底牌不能看到，只能看到背面或数量。
- 当前轮到自己行动时，会带上合法动作列表。

这个模块是防止信息泄露的关键层。

### history：当前手牌临时事件

```text
src/mcx_poker/history/
└── events.py
```

人话版：

- 这里记录当前手牌发生过什么。
- 比如新手牌开始、公共牌更新、玩家成功动作、非法动作、结算。
- MVP 里只是内存日志，不写数据库，不跨重启保存。
- 私有事件和公开事件分开，比如自己的底牌只能发给自己。

它以后可以演进成 Hand History、Replay、统计的数据基础。

### delivery：对外入口，API、WebSocket、前端

```text
src/mcx_poker/delivery/
├── api.py
├── websocket.py
├── local.py
└── static/
    ├── index.html
    ├── styles.css
    └── app.js
```

人话版：

- `api.py`：HTTP API。比如查桌子、开始、暂停、继续、重开。
- `websocket.py`：实时通道。前端通过它收到状态，也通过它提交真人动作。
- `local.py`：本地 MVP 的“组装器”。它把 Table Manager、Adapter、Game Loop、Controller、WebSocket 这些模块接起来，组成一个能跑的本地应用。
- `static/`：最小网页前端。

`static/index.html` 是页面结构，`styles.css` 是样式，`app.js` 是前端状态和 WebSocket 交互逻辑。

前端只展示后端给的数据，不自己算底池、不自己判断谁赢、不自己判断动作是否合法。

### extensions：未来扩展口子

```text
src/mcx_poker/extensions/
└── interfaces.py
```

人话版：

- 这里是给未来 LLM、GTO、Coach、Replay 留的接口。
- 现在只是占位和边界，不实现真实 LLM，不接 solver，不做 Coach 页面。
- 这么做是为了以后加功能时，不要把主 Game Loop 和 PokerKit Adapter 搞乱。

## tests：每个模块有没有跑通

```text
tests/
├── test_action_model.py
├── test_action_validator.py
├── test_pokerkit_adapter.py
├── test_table_manager.py
├── test_hand_event_log.py
├── test_observation_builder.py
├── test_player_controllers.py
├── test_game_loop.py
├── test_delivery_api.py
├── test_websocket_gateway.py
├── test_frontend_static.py
└── test_extension_interfaces.py
```

人话版：

- `tests/` 是安全网。
- 每个核心模块都有自己的测试。
- Adapter 测试会真的用 PokerKit。
- 其他模块多数用 fake/stub，避免一改 PokerKit 就影响所有测试。
- 前端测试主要检查静态页面和 JS 提交动作的关键约束。

最终结果是：

```text

```

那个 warning 来自 FastAPI/TestClient 依赖，不是业务失败。

## doc：项目为什么这么写

```text
doc/
├── .obsidian/
├── proposal.md
├── high-level-design.md
├── prompt.md
├── implementation-decisions.md
├── detailed-design/
├── tasks/
└── 人话.md
```

人话版：

- `.obsidian/`：如果你用 Obsidian 看文档，这里存的是 Obsidian 的界面配置。它不影响代码运行。
- `proposal.md`：需求来源和范围。
- `high-level-design.md`：整体架构怎么分层。
- `prompt.md`：这次实现 MVP 的执行说明。
- `implementation-decisions.md`：文档没完全确认的地方，我按 MVP 默认策略做了哪些决定。
- `人话.md`：就是这份，给人快速看懂项目结构。

### doc/.obsidian：文档工具配置

```text
doc/.obsidian/
├── app.json
├── appearance.json
├── core-plugins.json
└── workspace.json
```

人话版：

- 这些是 Obsidian 的配置文件。
- 它们只影响文档软件怎么显示、开什么插件、窗口布局是什么。
- 改业务、改测试、改架构时基本不用碰这里。

### doc/detailed-design：细分设计

```text
doc/detailed-design/
├── engine/
├── domain/
├── data/
├── delivery/
└── extensions/
```

人话版：

- `engine/`：动作、校验、PokerKit 适配器、Game Loop。
- `domain/`：牌桌、玩家控制器、Observation。
- `data/`：临时手牌事件日志。
- `delivery/`：API、WebSocket、前端 UI。
- `extensions/`：未来 LLM/GTO/Coach 接口。

这些是写代码前的设计依据。

### doc/tasks：任务清单

```text
doc/tasks/
├── progress.md
├── action-model.md
├── pokerkit-adapter.md
├── table-manager.md
└── ...
```

人话版：

- 每个模块一个任务文件。
- 里面有最小任务和测试验收项。
- 我已经把 MVP 已完成项勾上。

## 一条主线理解整个项目

如果用一句话串起来：

```text
前端点按钮
  -> WebSocket 收到动作
  -> Human Controller 交给 Game Loop
  -> Validator 先检查
  -> PokerKit Adapter 交给 PokerKit
  -> PokerKit 推进牌局
  -> Table Manager 更新长期桌子状态
  -> Observation 生成玩家视角
  -> WebSocket 推回前端
```

再强调一次边界：

- PokerKit 负责扑克规则。
- 后端是唯一权威。
- 前端只显示和提交。
- Observation 负责不泄露隐藏信息。
- Adapter 是唯一能碰 PokerKit 的业务模块。

## 现在没做的东西

这些不是遗漏，而是 MVP 边界外：

- 账号系统。
- 登录认证。
- 多桌大厅。
- 玩家离桌、补码、断线托管。
- 数据库。
- 持久化 Hand History。
- Replay UI。
- 统计系统。
- 真实 LLM 玩家。
- 真实 GTO solver。
- Coach 分析页面。
- 高级动画和美术级牌面。

## 最后一句人话

这个项目现在的形态是：一个本地能跑、能打开网页玩、后端管规则、PokerKit 管牌局、bot 能行动、真人能通过 WebSocket 操作的德州扑克 MVP。目录结构基本是按“规则核心、牌桌状态、玩家输入、可见信息、对外入口、未来扩展”这几块分开的。
